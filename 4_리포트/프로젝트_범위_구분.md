# 프로젝트 범위: 증명된 것 vs 향후 과제

> 발표 시 과장 없이 방어하기 위한 핵심 구분.
> "결과"와 "제안"을 명확히 분리한다.

---

## A. 본 연구에서 데이터로 증명한 것 (결과)

| # | 증명 내용 | 근거 |
|---|---|---|
| 1 | 부분 수면박탈은 PVT(객관 각성)에 둔감 | 짝비교 p>0.5 |
| 2 | 수면박탈은 주관 지표(SicknessQ)에는 반영됨 | d=0.52, p<0.001 |
| 3 | 주관(SicknessQ)과 객관(PVT)은 독립된 축 | 상관 r≈0, χ² p=0.143 |
| 4 | 졸림 척도(KSS)는 PVT와 연관 (SicknessQ와 대조) | DROZY r=0.63 |
| 5 | "숨은 위험"(각성 저하를 본인이 모름)이 실재 | 27.8%, 47세션 |
| 6 | 개인특성으로 주관 피로 예측 가능 | AUC 0.76~0.81 |
| 7 | 집단 기준은 "원래 느린 사람"을 위험으로 오인 | 숨은위험 47 중 37이 개인차 |

→ **결론: PVT + 주관 지표 + 개인특성으로 피로 상태를 2×2로 분류할 수 있다.**
   이것이 현재 데이터로 구현 가능한 서비스의 범위.

---

## B. 향후 서비스 운영으로 검증할 것 (제안·미검증)

| # | 제안 내용 | 왜 지금 증명 불가한가 |
|---|---|---|
| 1 | 개인 baseline 축적 → z-score로 개인차 제거 | 1인 2세션뿐, 반복측정 부족 |
| 2 | baseline 확립 후 판정 정확도 향상 | 장기 반복측정 데이터 필요 |
| 3 | 사용할수록 개인화되어 성능 개선 | 실제 운영 데이터로만 확인 가능 |

→ 이는 **근거 없는 기대가 아니라**, A-7(집단 기준의 한계)에서
   논리적으로 도출된 **근거 있는 설계 제안**이다.
→ 단, 효과의 정량 검증은 실제 서비스 운영 데이터를 통한 향후 과제.

## C. 정직한 서술 원칙

- **A는 "결과"로**, **B는 "향후 방향"으로** 명확히 구분해 서술
- 과장 예시(지양): "우리 서비스는 쓸수록 정확해진다"
- 정직 예시(권장): "개인차가 크다는 것을 데이터로 확인했고(A-7),
  이에 개인 baseline 기반 z-score를 제안한다. 그 성능 개선 효과는
  운영 데이터를 통한 향후 검증 과제다."
